# Rusty due to using LLMs

Honest answer first: you're not unique in this. Most working developers in 2024-2025 have drifted the same way.
The goal isn't to stop using AI — it's to be able to work without  it when you need to, and to actually understand
what it's generating so you can catch when it's wrong.

---
The core problem to fix

You don't have a knowledge gap, you have a recall gap. You've seen this code hundreds of times, you just can't produce
it under pressure. That's a different problem than not knowing it — and it has a specific fix: regular low-stakes
production.

---
The sustainable routine

Daily (15 minutes, non-negotiable)

Pick one of these, rotate:
- Open a blank file and rebuild one exercise from this repo from memory
- Go to exercism.org (free, has a JavaScript track) and do one exercise — they're small and well-scoped
- Read one page of javascript.info — the best JS reference that exists, free, reads fast

That's it. 15 minutes. The consistency matters more than the volume.

Weekly (1 hour, one session)

Add a new exercise to this repo. The source doesn't matter — something you ran into at work, an interview problem you
heard about, a feature you built with AI that you didn't fully understand. Write it from scratch yourself.

This repo becomes your long-term record of problems you've worked through. In a year it'll have 50+ exercises and
you'll be able to rebuild any of them.

Monthly (2 hours)

Do a mock interview. Either:
- Ask me to give you a timed coding challenge and do it live in the chat
- Use interviewing.io (free practice interviews) or Pramp
- Just set a 25-minute timer and build something cold

---
Topics to cover in order

Don't try to cover everything at once. Work through these over 6 months:

Months 1-2 — JS fundamentals (what this repo covers)
- Array methods: filter, map, reduce, find, sort
- String manipulation
- Objects and destructuring
- async/await and fetch

Months 3-4 — React patterns
- useState, useEffect, useRef
- Lifting state up
- Fetching data in components
- Controlled forms

Months 5-6 — Light algorithms
- Binary search, linear search
- Sorting basics (don't memorize implementations, understand when/why)
- Working with nested data (arrays of objects, grouping, aggregating)

You don't need LeetCode hard problems. 90% of real interviews at the level BairesDev is testing are array manipulation
+ DOM or React. The algorithm stuff is a bonus.

---
The AI rule going forward

Use AI the way you use a calculator — for things you already understand, to go faster. The test is: could you explain to
someone exactly what the AI wrote and why? If no, rewrite it yourself until you can.

Specifically:
- Let AI generate boilerplate, scaffolding, config
- Write the core logic yourself first, then compare with what AI would write
- When AI generates something you don't fully understand, ask it to explain it, then close the explanation and rewrite
  it from scratch

MDN is the bible mid-interview. The pattern is always the same: you know the method exists, you just forget the exact
signature or what it returns. MDN answers that in 5 seconds. Bookmark the Array page above specifically, you'll use it
every single session.

The honest timeline: if you do 15 minutes a day consistently, in 3 months you'll never feel this unprepared again. In
6 months writing code by hand will feel normal. The people who stay sharp aren't smarter, they just never fully stopped
writing things from scratch.
