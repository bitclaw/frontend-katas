---
Phase 1: Before the interview (priority)

Day 1  Just read and understand, don't write yet

Open each exercise in the browser, read the code, make sure you can explain every line out loud. Don't skip this,
understanding beats memorizing.

Order:
1. 01-realtime-search — this is the actual interview
2. 03-debounced-search — likely follow-up
3. 02-task-manager — the second exercise they described

Day 2 — Write from scratch, no looking

Close the index.html files. Open a blank file. Build each one from memory. It will be painful at first — that's the
point. When you get stuck, look at the cheat sheet in the main README, not the solution.

Repeat until 01 feels boring.

Day 3 — Simulate the interview

Set a timer for 20 minutes. Screen-share with yourself (or just pretend). Build 01-realtime-search from scratch while
narrating what you're doing out loud. If you can explain it while writing it, you're ready.

---
Phase 2: After the interview (ongoing)

One exercise per session, two or three times a week. The workflow for each:

1. Read the README — understand the problem
2. Try to build it from scratch without looking at index.html
3. When stuck, check the cheat sheet first, then the solution
4. Once it works, delete your code and do it again

The second time through any exercise should take half the time of the first. If it doesn't, do it a third time.

---
How to add new exercises

When you encounter a new problem in an interview or online:
1. Make a new folder (09-whatever/)
2. Write the README.md first — problem statement + what to focus on
3. Build the solution in index.html
4. Come back to it blank a week later

The act of writing the README forces you to understand the problem before you solve it, which is exactly what you need
to do in a live interview.

---
The one rule

Never run code you haven't written. Copy-pasting from the solutions and tweaking doesn't build the muscle memory you
need. The discomfort of staring at a blank file is the whole point.
